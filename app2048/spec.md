# 2048 게임 — spec.md (Plan 산출물)

> 사용자 요청: "2048게임을 만들어줘. 방향키로 숫자 타일을 밀어서 합치는 퍼즐 게임이야. 점수판도 넣어줘."
> 상태: 계획 단계 — 사용자 승인 대기. 승인 전까지 `/app2048/` 안에는 `PLAN.md`, `spec.md`만 존재한다.

## 1. 앱 개요

**2048**은 4x4 격자에서 방향키(↑↓←→)로 모든 타일을 한 방향으로 밀어, 같은 숫자 타일끼리 만나면 합쳐 점수를 올리는 싱글플레이어 퍼즐 게임이다.

핵심 규칙:
- 보드는 4x4 = 16칸. 각 칸은 비어있거나 2의 거듭제곱 숫자 타일(2, 4, 8, … )을 가진다.
- 게임 시작 시 무작위 빈 칸 2곳에 타일(2 또는 4)을 생성한다.
- 방향키를 누르면 모든 타일이 그 방향 끝까지 밀리며, 이동 경로상 같은 숫자의 타일 두 개가 만나면 하나로 합쳐지고 값이 2배가 된다(한 번의 이동에서 한 타일은 최대 한 번만 병합됨 — 연쇄 병합 없음).
- 병합으로 생긴 새 값만큼 점수에 가산된다(예: 4+4→8 병합 시 +8점).
- 실제로 보드 상태가 바뀐 이동(타일이 하나라도 움직이거나 합쳐진 경우)에만 빈 칸 중 하나에 새 타일(2 또는 4, 확률 90%/10%)이 생성된다. 보드가 전혀 안 바뀌는 방향키 입력(막힌 벽 쪽으로 밀기 등)은 무시하고 새 타일도 생성하지 않는다.
- 2048 값의 타일이 처음 만들어지면 **승리** 상태이며, 승리 오버레이를 띄운다. 사용자는 "계속하기"로 더 큰 타일(4096 등)에 도전하거나 "새 게임"으로 재시작할 수 있다.
- 보드가 가득 찼고(빈 칸 없음) 상하좌우 어느 방향으로도 인접한 두 타일이 합쳐질 수 없으면 **게임 오버**이며, 게임 오버 오버레이를 띄운다.

## 2. 화면 구성

이 게임은 **단일 화면**으로 설계한다:
- 헤더(타이틀 "2048" + 짧은 설명)
- 점수판(SCORE 현재 점수 / BEST 최고 기록)
- "새 게임" 버튼
- 4x4 보드(타일 렌더링 영역)
- 승리/게임오버 오버레이(보드 위에 겹쳐서 나타나는 모달형 패널, 별도 라우트/페이지 아님)

화면 전환이나 별도 라우팅이 없고 모든 상태(플레이 중 / 승리 / 게임오버)가 같은 DOM 안에서 오버레이 표시 여부로만 갈리므로, **CLAUDE.md 기준상 "화면이 3개 미만"에 해당한다. 따라서 Build 서브에이전트를 화면별로 나눌 필요가 없다** — Build는 단일 서브에이전트, 단일 `BUILD.md`로 진행한다.

## 3. 파일 구조

```
/app2048/
  spec.md         # 본 파일 (Plan 산출물)
  BUILD.md        # Build 서브에이전트 지침 (사용자 승인 후 별도로 생성 예정)
  review.md       # Review 서브에이전트 산출물 (Build 이후 별도 단계)
  index.html      # 앱 진입점 — 마크업 뼈대(헤더/점수판/보드/오버레이), styles.css·game.js·app.js 로드
  styles.css      # 이 앱 전용 스타일 (블로그 tokens.css를 참고하여 값 복제/조정 — 실시간 import 아님, 자체 완결)
  game.js         # 순수 게임 엔진 모듈: 보드 상태, 이동/병합 로직, 승리·게임오버 판정 (DOM 비의존)
  app.js          # UI 컨트롤러: game.js를 불러와 DOM 렌더링, 키보드/터치 입력 처리, localStorage 최고점수 연동
```

- **외부 라이브러리/CDN 없음.** 2048의 로직(배열 이동/병합, 랜덤 타일 생성)은 프레임워크 없이 순수 JS로 충분히 구현 가능하며, CLAUDE.md의 "외부 라이브러리 사용 최소화" 원칙과 "자체 완결" 원칙에 가장 부합한다. 폰트도 블로그처럼 시스템 폰트 스택을 그대로 사용해 웹폰트 CDN 요청 자체를 없앤다.
- `game.js`(엔진)와 `app.js`(UI/입력) 분리는 블로그 기존 코드의 관례(`js/frontmatter.js`·`js/markdown.js` = 순수 로직 모듈 vs `js/list-page.js`·`js/post-page.js` = 페이지 컨트롤러)를 그대로 따른 것이다.
- `styles.css`는 `styles/tokens.css`의 값(색상 hex, spacing, radius 스케일)을 **참고해 앱 폴더 안에 자체 변수로 복제**한다. `/app2048/`이 블로그 루트 CSS 파일을 직접 `<link>`로 가져오면 "자체 완결" 원칙(앱마다 독립 폴더, 블로그 다른 파일에 의존하지 않음)에 어긋나므로, 값만 참고하고 파일은 복제한다.

## 4. 게임 로직 설계 요약

### 4.1 상태 표현
- `board`: `number[4][4]`, 0은 빈 칸, 그 외 값은 타일 숫자(2, 4, 8, …).
- `score`: 현재 점수(숫자).
- `best`: 최고 점수(숫자, localStorage에서 로드).
- `won`: 2048 타일을 이미 달성했는지(승리 오버레이 재노출 방지 및 "계속하기" 처리용 플래그).
- `over`: 게임 오버 여부.

### 4.2 이동+병합 알고리즘(방향 공통 절차)
4방향(상/하/좌/우) 모두 같은 절차를 "라인(행 또는 열) 4개"에 대해 반복 적용한다:
1. 방향에 따라 보드를 라인 배열로 추출한다.
   - 좌(←): 각 행을 그대로.
   - 우(→): 각 행을 뒤집어서.
   - 상(↑): 각 열을 위→아래 순서로.
   - 하(↓): 각 열을 아래→위 순서로.
2. **압축**: 라인에서 0을 제거해 값들을 진행 방향 앞쪽으로 모은다.
3. **병합**: 앞에서부터 순회하며 `line[i] === line[i+1]`이면 `line[i] *= 2`로 합치고 `score += line[i]`, `line[i+1]`은 제거(스킵), 한 타일은 이번 이동에서 최대 한 번만 병합되도록 병합된 인덱스는 다시 비교하지 않는다.
4. 병합 후 다시 압축하고 남는 칸은 0으로 채워 길이 4를 유지한다.
5. 1번의 역변환으로 원래 보드 좌표에 다시 써넣는다(우: 다시 뒤집기, 상/하: 다시 전치).
6. 이동 전/후 보드를 비교해 하나라도 달라졌으면 `moved = true`.

### 4.3 새 타일 생성
- `moved === true`인 경우에만 빈 칸 중 하나를 무작위로 골라 2(90%) 또는 4(10%)를 생성한다.
- 게임 시작/새 게임 시에는 위 규칙과 별개로 빈 보드에 타일 2개를 생성한다.

### 4.4 점수 규칙
- 병합이 일어날 때마다 **병합으로 생긴 새 값**만큼 `score`에 가산한다(2048 원작 규칙과 동일).
- 매 이동 후 `score > best`이면 `best = score`로 갱신.

### 4.5 최고 점수 저장 (localStorage)
- 키 제안: `app2048-best-score` (블로그의 `theme` 키와 충돌 없는 앱 전용 네임스페이스).
- "새 게임"을 눌러도 `best`는 초기화하지 않는다(현재 `score`만 0으로 리셋). 페이지를 새로고침해도 `best`는 유지된다.
- 별도의 "최고점수 초기화" 버튼은 요구사항에 없으므로 이번 스펙에서는 만들지 않는다.

### 4.6 승리/게임오버 판정
- 승리: 보드 내 어떤 칸이든 값이 2048이 되고 `won === false`일 때 최초 1회 승리 오버레이 표시, 이후 `won = true`로 전환. "계속하기"를 누르면 오버레이만 닫고 게임은 계속(더 큰 타일 도전 가능), "새 게임"을 누르면 완전 초기화(이때 `won`도 false로 리셋).
- 게임 오버: 빈 칸이 0개이고, 모든 인접 쌍(가로/세로)에 대해 값이 같은 쌍이 하나도 없을 때 게임 오버 오버레이 표시. 이 상태에서는 방향키/스와이프 입력을 무시(또는 오버레이의 "다시 시작" 버튼만 활성).

## 5. 조작 방식

- **키보드(필수)**: `window`에 `keydown` 리스너, `ArrowUp/Down/Left/Right` 처리 + `preventDefault()`로 페이지 스크롤 방지. 오버레이(승리/게임오버)가 떠 있는 동안은 이동 입력을 무시한다.
- **모바일 터치 스와이프(필수 — CLAUDE.md 웹앱 규칙 "모바일에서도 사용할 수 있어야 한다" 충족)**: 보드 요소에 `touchstart`/`touchend` 좌표 차이를 계산해 스와이프 방향 판정(임계값 예: 30px 이상, 더 큰 축을 방향으로 채택). 보드 영역에서만 `touchmove` 기본 동작을 제한해 스와이프 중 페이지 스크롤과 충돌하지 않도록 한다. 그 외 영역(헤더, 버튼)은 일반 터치 스크롤/탭 그대로 허용.
- "새 게임" 버튼과 오버레이의 "다시 시작"/"계속하기" 버튼은 마우스 클릭과 터치 탭 모두 동작(표준 `click` 이벤트로 충분).

## 6. 디자인 톤

`DESIGN-airtable.md`와 `styles/tokens.css`를 참고해 **게임을 감싸는 UI 크롬(헤더, 점수판, 버튼, 오버레이 카드, 페이지 배경)은 블로그 톤을 유지**하고, **보드/타일 자체는 2048 특유의 팔레트를 허용**한다.

- 페이지 배경: `--color-canvas`(라이트 `#ffffff` / 다크 `#14161b`), 본문 텍스트: `--color-ink`/`--color-body`.
- "새 게임" 버튼: 블로그의 `button-primary` 패턴을 그대로 차용 — near-black(`--color-primary`) 배경, 흰 텍스트, `rounded-lg`(12px), hover 상태 없음(no-hover 정책 준수, active/pressed 상태만 정의).
- 점수판 박스(SCORE/BEST): `rounded-md`(10px), 옅은 서피스 배경.
- 보드 자체: 2048 고유 팔레트 사용 — 보드 배경은 따뜻한 회갈색(`#bbada0` 계열), 타일은 값별 전통 2048 색상 계단(2→`#eee4da`, 4→`#ede0c8`, 8→`#f2b179`, …, 2048→`#edc22e` 등)을 채택. 타일 코너는 블로그 radius 스케일 중 `rounded-sm`~`rounded-md`로 통일해 완전히 이질적으로 보이지 않게 한다.
- 승리/게임오버 오버레이: 블로그의 signature-card 패턴 차용 — 승리는 밝은 크림/서피스 톤 카드, 게임 오버는 `surface-dark`(near-black) 카드에 흰 텍스트, 둘 다 `rounded-lg`.
- **다크모드**: `prefers-color-scheme: dark`를 기본 신호로 앱 자체 CSS 안에서 대응(값은 블로그 `styles/tokens.css`의 다크 오버라이드를 참고해 복제). 이 앱은 블로그의 전역 다크모드 토글(`js/theme.js`)을 공유하지 않으므로(자체 완결 원칙), OS 설정 자동 반영만 기본으로 하고 별도 토글 버튼은 만들지 않는다. 타일 팔레트는 다크모드에서도 원작 2048 색감을 유지하고 보드 바깥 크롬만 반전한다.
- 접근성/터치 타깃: 버튼 44×44px 이상 확보. 키보드 조작 게임이므로 포커스 아웃라인은 유지한다.
