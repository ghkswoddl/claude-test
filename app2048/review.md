# review.md — 2048 게임 Review 결과

독립 Review 서브에이전트가 `http://localhost:8080/app2048/index.html`을 실제 브라우저(Browser pane)에서 구동하며 검증했다. Build의 자체 점검을 신뢰하지 않고, 코드 리딩 + 키보드/터치 이벤트 디스패치 + DOM/localStorage/네트워크/콘솔 직접 조사로 독립 검증했다.

## 검증 환경 관련 참고사항

Browser pane의 `computer{screenshot}` 액션과 `read_page` 접근성 스냅샷 도구가 이 세션에서 지속적으로 타임아웃/스테일 데이터를 반환했다(`read_page`가 실제 라이브 DOM과 무관한 과거 상태 — "승리!" 오버레이 + BEST 664 — 를 계속 보고함). `javascript_tool`로 라이브 DOM(`document.getElementById(...).textContent`, `overlay.hidden` 등)을 직접 조회하면 실제 상태(오버레이 hidden:true, SCORE 0)와 다름을 확인해 이것이 **툴링 문제**임을 검증했고, 이후 모든 검증은 `computer{key}`(실제 키 입력)와 `javascript_tool`을 통한 `KeyboardEvent`/`TouchEvent` 디스패치 + DOM/상태 직접 조회로 진행했다. 앱 자체의 결함이 아님을 명확히 하기 위해 기록한다.

## 검증 항목 및 결과

## 발견한 문제와 수정 내역

### 버그: 다크모드에서 승리 오버레이 텍스트가 거의 안 보임 (수정 완료)

- **파일**: `app2048/styles.css`
- **증상**: `.overlay-card-light`(승리 오버레이의 시그니처 크림 카드)가 `color: var(--color-ink)`를 사용하고 있었음. `--color-ink`는 다크모드에서 `#f2f3f5`(거의 흰색)로 바뀌는 반면, 배경으로 쓰인 `--color-signature-cream`(`#f5e9d4`)은 CLAUDE.md의 "시그니처 카드는 캔버스 반전의 영향을 받지 않는다" 원칙에 따라 다크모드에서도 값이 고정됨. 그 결과 다크모드에서 **크림색(밝은) 배경 위에 거의 흰색 텍스트**가 겹쳐, 실측 대비율이 **1.08:1**(WCAG AA 최소 4.5:1에 크게 미달, 사실상 읽기 불가능)로 나왔음.
- **재현 방법**: `prefers-color-scheme: dark`로 설정 후 보드를 1024+1024 인접 상태로 조작해 승리를 유도, `overlay-title`의 `getComputedStyle().color`와 `overlay-card`의 `backgroundColor`로 WCAG 대비 공식을 직접 계산해 확인.
- **수정**: `.overlay-card-light`의 `color`를 테마에 따라 바뀌는 `var(--color-ink)` 대신 고정된 어두운 잉크색 `#181d26`으로 변경(시그니처 카드는 자체 배경을 갖고 있으므로 전경색도 테마와 무관하게 고정되어야 함). 수정 후 재측정 결과 라이트/다크 모드 모두 대비율 **14.07:1**로 개선됨을 확인.
- 게임오버 오버레이(`.overlay-card-dark`, `--color-surface-dark` + `--color-on-dark`)는 두 값 모두 라이트/다크 모드에서 동일한 hex(`#181d26`/`#ffffff`)로 정의되어 있어 원래부터 문제없음(대비율 16.9:1로 확인).

## 수정하지 않고 참고만 한 사항 (스펙 준수 범위 내 특성)

- **타일 8/16 등 중간 값 타일의 텍스트 대비가 낮음**(실측: `tile-8` 약 1.72:1, `tile-16` 약 2.08:1). 이는 spec.md 6장이 명시한 "전통 2048 색상 계단"(`#f2b179` 등 오리지널 2048 팔레트)을 그대로 채택한 결과이며, 오리지널 2048 게임 자체도 동일한 특성을 가진다(원작은 폰트 굵기 700으로 다소 보완하나 이 프로젝트는 CLAUDE.md의 "600 이상 굵기 금지(레이블/배너 제외)" 원칙 때문에 500을 사용). 라이트/다크 모드 공통이며 UI 크롬 대비 요구사항과는 별개의 보드 고유 팔레트 영역이라 스펙을 벗어나는 임의 수정은 하지 않았다. 필요 시 후속 개선 항목으로 남긴다.

## 최종 결론

핵심 게임 로직(이동/병합/점수/신규타일 생성/승리·게임오버 판정), 키보드·터치 입력, "새 게임"/오버레이 버튼 동작, localStorage 최고점수 저장/유지, 반응형 레이아웃(320px~1440px, 가로 스크롤 없음), 콘솔 에러 없음, 외부 CDN 요청 없음(전부 200 OK) 모두 실제 브라우저 상호작용으로 검증 완료. 다크모드에서 발견된 승리 오버레이 텍스트 대비 버그는 `app2048/styles.css` 한 곳을 수정해 해결하고 재검증까지 마쳤다.

**배포 가능**: 예. 추가로 발견된 블로킹 이슈 없음.
