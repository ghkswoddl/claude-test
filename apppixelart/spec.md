# 픽셀 아트 에디터 — spec.md (Plan 산출물)

> 사용자 요청: "픽셀아트에디터를 만들어줘. 16*16격자에 도트를 찍어서 그림을 그리고 완성하면 png로 저장할 수 있게 해줘. 색상 팔레트도 넣어줘."
> 상태: 계획 단계 — 사용자 승인 대기. 승인 전까지 `/apppixelart/` 안에는 `PLAN.md`, `spec.md`만 존재한다.

## 1. 앱 개요

**픽셀 아트 에디터**는 16×16 격자 위에 클릭(및 드래그)으로 점을 찍어 그림을 그리는 싱글 화면 드로잉 도구다.

핵심 기능:
- 16×16 = 256칸의 격자. 각 칸은 비어있음(투명) 또는 사용자가 선택한 색으로 채워짐.
- 색상 팔레트에서 기본 제공 색 중 하나를 클릭해 "현재 색"으로 선택하고, 격자 칸을 클릭하면 그 칸이 현재 색으로 칠해진다.
- 마우스(또는 손가락)를 누른 채로 격자 위를 이동하면 지나가는 칸이 연속으로 칠해진다(드래그 페인팅).
- 지우개 도구로 특정 칸을 다시 투명 상태로 되돌릴 수 있고, "전체 지우기" 버튼으로 격자를 한 번에 초기화할 수 있다.
- "PNG로 저장" 버튼을 누르면 그린 그림을 PNG 파일로 다운로드한다(투명 배경 유지, 확대 해상도 — 이유는 4.5절).
- 페이지 새로고침 시 그림이 유지되는 영속 저장(localStorage 등)은 사용자 요청 범위에 없으므로 이번 스펙에 포함하지 않는다. 필요해지면 별도 요청으로 추가한다.

## 2. 화면 구성

이 앱은 **단일 화면**으로 설계한다:
- 헤더(타이틀 "픽셀 아트 에디터" + 짧은 설명)
- 툴바(펜/지우개 토글, 전체 지우기, PNG로 저장 버튼)
- 색상 팔레트 패널(기본 색상 스와치 + 커스텀 색상 피커)
- 16×16 격자(그리기 작업 공간)

모든 상태(그리기 중 / 지우개 모드)가 같은 DOM 안에서 버튼 토글과 클릭 핸들러만으로 갈리고, 별도 라우팅이나 모달 페이지 전환이 없다. 화면은 1개이므로 **CLAUDE.md 기준상 "화면이 3개 미만"에 해당한다. 따라서 Build 서브에이전트를 화면별로 나눌 필요가 없다** — Build는 단일 서브에이전트, 단일 `BUILD.md`로 진행한다.

## 3. 파일 구조

```
/apppixelart/
  spec.md         # 본 파일 (Plan 산출물)
  BUILD.md        # Build 서브에이전트 지침 (사용자 승인 후 별도로 생성 예정)
  review.md       # Review 서브에이전트 산출물 (Build 이후 별도 단계)
  index.html      # 앱 진입점 — 마크업 뼈대(헤더/툴바/팔레트/격자), styles.css·grid.js·editor.js 로드
  styles.css      # 이 앱 전용 스타일 (블로그 tokens.css 값을 참고하여 복제 — 자체 완결, import 아님)
  grid.js         # 순수 데이터 모델: 16x16 셀 배열, get/set/clear 픽셀 (DOM·canvas 비의존)
  editor.js       # UI 컨트롤러: canvas 렌더링, 포인터(마우스/터치) 입력 처리, 팔레트·툴바 이벤트, PNG 내보내기
```

- **외부 라이브러리/CDN 없음** — 근거는 5절.
- `grid.js`(데이터 모델)와 `editor.js`(UI/입력/렌더링) 분리는 app2048의 관례(`game.js` = 순수 로직 vs `app.js` = UI 컨트롤러, 더 거슬러 올라가면 블로그 `js/frontmatter.js`·`js/markdown.js` vs `js/list-page.js`·`js/post-page.js`)를 그대로 따른 것이다. 픽셀 격자 데이터 모델 자체는 2048 엔진보다 훨씬 단순하지만("칸에 색 있음/없음"이 전부), 그래도 "격자가 지금 어떤 상태인가"라는 질문을 canvas·DOM 렌더링 코드와 분리해두면 `editor.js`가 렌더링/입력 처리에만 집중할 수 있어 관례를 유지한다.
- `styles.css`는 `styles/tokens.css`의 값(색상 hex, spacing, radius 스케일)을 **참고해 앱 폴더 안에 자체 변수로 복제**한다. app2048과 동일한 이유 — 앱마다 독립 폴더로 자체 완결해야 하므로 블로그 루트 CSS를 직접 `<link>`하지 않는다.

## 4. 핵심 기능 설계

### 4.1 16×16 격자 렌더링 방식 — `<canvas>` 채택

**DOM 기반(256개 `<div>` 격자) vs `<canvas>` 두 방식을 비교해 `<canvas>`를 채택한다.**

- **PNG 저장이 필수 요구사항**이므로 최종적으로는 반드시 `<canvas>`의 `toDataURL('image/png')`(또는 `toBlob`)을 거쳐야 한다. DOM 격자로 그리는 방식을 택하더라도 저장 시점에는 결국 색상 배열을 다시 canvas에 그려 넣어 `toDataURL`을 호출해야 하므로, "그리기용 표현"과 "저장용 표현"이 이원화된다.
- 이 앱은 **처음부터 canvas 하나를 그리기 작업 공간이자 저장 소스로 겸용**한다: 사용자가 클릭하는 canvas가 곧 내보낼 PNG의 원본이므로 별도 변환 단계가 없고, "화면에 보이는 것 = 저장되는 것"이 보장된다.
- **레이어 구성(같은 위치에 absolute로 겹침, 아래→위 순서)**:
  1. `checkerboard-bg` — canvas 밑에 깔리는 CSS 배경(회색/흰색 체크무늬, `repeating-conic-gradient` 등으로 구현). 투명 칸을 시각적으로 구분하기 위한 용도이며 **내보내기 대상이 아니다**.
  2. `pixel-canvas` — 실제로 칠한 색이 저장되는 canvas. 배경을 채우지 않고 시작해 기본 상태가 완전 투명이며, 지우개는 `clearRect`로 해당 칸을 다시 투명하게 만든다. **PNG 저장은 이 canvas 하나만 `toDataURL`한다.**
  3. `grid-overlay` — 격자선만 그리는 별도 canvas, `pointer-events: none`. 편집 중 칸 구분을 돕는 순수 시각 요소이며 `pixel-canvas`와 분리되어 있으므로 **내보낸 PNG에는 격자선이 절대 섞이지 않는다.**
- 클릭/드래그 이벤트는 `pixel-canvas`에 직접 바인딩한다(위 overlay가 `pointer-events: none`이므로 이벤트가 자연스럽게 통과해 `pixel-canvas`가 받는다).
- **DOM 격자 방식의 트레이드오프(채택하지 않은 이유)**: 256개 `<div>`는 칸별 클릭 핸들러 연결이 직관적이라는 장점이 있지만 (a) 저장 시 결국 canvas로 재변환하는 이중 작업이 필요하고, (b) 격자선/칸 스타일을 CSS로 관리해야 해 상태(색상)와 표현(DOM)이 분리되어 동기화 버그 여지가 있으며, (c) 모바일에서 드래그 페인팅 시 "지금 손가락이 어느 div 위에 있는가"를 `elementFromPoint` 등으로 다시 계산해야 해 canvas 좌표 계산보다 오히려 더 번거롭다. 따라서 이번 스펙에서는 canvas 단일 방식으로 확정한다.

### 4.2 클릭 + 드래그로 연속 칠하기

- 마우스와 터치를 통합해 다루는 **Pointer Events**(`pointerdown` / `pointermove` / `pointerup` / `pointercancel`)를 `pixel-canvas`에 바인딩한다. 별도의 마우스 전용/터치 전용 핸들러를 각각 만들지 않고 하나의 이벤트 체계로 처리해 코드 중복을 줄인다(대부분의 최신 브라우저 및 iOS Safari 13+ 지원).
- `pointerdown`: 눌린 지점의 칸 좌표를 계산해 즉시 1칸 칠하고(또는 지우개 모드면 지움), 이후 이동을 추적하기 위해 `canvas.setPointerCapture(event.pointerId)` 호출.
- `pointermove`(포인터가 눌려있는 동안만): 이동한 지점의 칸 좌표를 계산 → 직전에 칠한 칸과 다르면 새 칸을 칠한다(같은 칸 안에서의 자잘한 이동은 무시해 불필요한 재렌더링을 줄임).
- `pointerup` / `pointercancel`: 드래그 상태 종료.
- **좌표 → 칸 인덱스 변환**: `canvas.getBoundingClientRect()`로 실제 화면 표시 크기를 구하고, `scaleX = canvas.width / rect.width`(캔버스 내부 해상도 ÷ CSS 표시 크기)로 비율을 보정한 뒤 `cellSize = canvas.width / 16`으로 나눠 칸 인덱스를 구한다. 이렇게 하면 모바일처럼 canvas가 CSS로 축소 표시되는 경우에도 클릭 위치가 정확한 칸에 매핑된다.
- 격자 경계 밖으로 포인터가 나가면(`pointermove` 좌표가 0~15 범위를 벗어나면) 해당 이벤트는 무시한다(에러 없이 안전하게 클램프 또는 스킵).

### 4.3 색상 팔레트

- **기본 제공 색상 16개**를 스와치 그리드로 배치한다. 픽셀아트 도구에서 널리 쓰이는 PICO-8 계열의 대비 좋은 16색 팔레트를 채택(검정·크림 화이트 포함, 저채도/고채도가 고르게 섞여 실제 그림을 그리기 좋음):

  | 스와치 | HEX | 스와치 | HEX |
  |---|---|---|---|
  | 검정 | `#000000` | 빨강 | `#FF004D` |
  | 다크 네이비 | `#1D2B53` | 주황 | `#FFA300` |
  | 자주 | `#7E2553` | 노랑 | `#FFEC27` |
  | 다크 그린 | `#008751` | 연두 | `#00E436` |
  | 브라운 | `#AB5236` | 하늘색 | `#29ADFF` |
  | 다크 그레이 | `#5F574F` | 인디고 | `#83769C` |
  | 라이트 그레이 | `#C2C3C7` | 핑크 | `#FF77A8` |
  | 크림 화이트 | `#FFF1E8` | 살구색 | `#FFCCAA` |

- **커스텀 색상**: 팔레트 맨 끝에 "커스텀" 스와치를 하나 두고, 클릭 시 숨겨진 네이티브 `<input type="color">`를 트리거(`label[for]` 또는 `.click()`)한다. 사용자가 색을 고르면 그 값이 즉시 "현재 색"이 되고 커스텀 스와치 자체의 미리보기 색도 갱신된다. 외부 컬러피커 라이브러리 없이 브라우저 네이티브 기능만으로 구현 가능하다.
- 스와치를 클릭하면 (a) 해당 색이 "현재 색"이 되고, (b) 도구 모드가 자동으로 "펜"으로 전환된다(지우개가 켜져 있던 상태에서 팔레트를 클릭하면 바로 그 색으로 그릴 수 있어야 자연스러움). 현재 선택된 스와치는 테두리 강조로 표시.
- 팔레트에 "투명" 스와치를 별도로 넣지 않는다 — 투명 처리는 아래 4.4절의 지우개 도구가 전담하며, 팔레트에도 동일한 기능을 중복 배치하면 "지우개 버튼과 투명 스와치 중 뭘 눌러야 하지"라는 혼동을 줄 수 있어 단일 진입점으로 통일한다.

### 4.4 지우개 도구 / 전체 지우기

- 툴바에 "지우개" 토글 버튼을 둔다. 활성화되면 이후의 클릭/드래그는 색을 칠하는 대신 `clearRect`로 해당 칸을 투명하게 되돌린다. 버튼은 활성 상태를 시각적으로 표시(예: 눌림 스타일)한다.
- "전체 지우기(Clear)" 버튼: 격자 전체(`pixel-canvas` 전체)를 `clearRect`로 비우고 데이터 모델(`grid.js`)의 배열도 전부 투명 값으로 재설정한다. 실수 클릭 방지를 위해 네이티브 `confirm()` 다이얼로그로 한 번 확인받는다(그림을 다 그린 뒤 실수로 날리는 것을 막는 최소한의 안전장치, 별도 UI 라이브러리 불필요).

### 4.5 PNG 저장 — 512×512(32px/칸) 확대 저장 채택

- **16×16 원본 해상도 그대로 저장하지 않는다.** 16×16 PNG는 대부분의 이미지 뷰어/공유 환경에서 너무 작게 보이거나 흐릿하게 확대되어 픽셀아트 특유의 "각진 도트" 느낌이 사라지기 쉽다.
- 대신 **확대 저장을 기본 정책으로 채택**한다. 4.1절의 레이어 설계에 따라 `pixel-canvas`의 내부 해상도(`canvas.width`/`canvas.height`)를 처음부터 512×512(칸당 32px)로 만들고, 매 칸을 `fillRect(col*32, row*32, 32, 32)`로 정확한 픽셀 경계에 맞춰 채운다. 이 방식은 "16×16으로 그린 뒤 저장 시점에 별도로 확대(scale-up) 처리"하는 2단계 과정이 필요 없다 — **그리는 canvas 자체가 이미 최종 내보내기 해상도**이므로 nearest-neighbor 확대 로직이나 이미지 스무딩 끄기(`imageSmoothingEnabled = false`) 같은 별도 처리 없이도 각 칸이 처음부터 뭉개지지 않는 32×32px 사각형으로 그려진다.
- "저장" 버튼 클릭 시: `pixelCanvas.toDataURL('image/png')` → 결과를 `<a download="pixel-art.png">` 임시 링크에 연결해 클릭 트리거하는 표준 방식으로 다운로드한다(사용자 액션에서 즉시 실행되므로 팝업 차단 이슈 없음).
- 배경을 한 번도 `fillRect`로 채우지 않고 시작하므로 `toDataURL('image/png')` 결과에는 투명 배경이 그대로 보존된다(완료 이후 배경이 투명한 픽셀아트 스프라이트로 바로 활용 가능).
- 여러 해상도 옵션(원본 16×16 / 확대 512×512 선택 UI 등)은 요청 범위를 벗어나는 과설계이므로 이번 스펙에서는 만들지 않는다 — 확대 저장 하나로 고정한다.

### 4.6 모바일 터치 지원 (CLAUDE.md 필수 요건)

- 4.2절의 Pointer Events 채택 자체가 마우스/터치를 하나의 코드 경로로 처리하므로 별도의 `touchstart`/`touchmove` 핸들러를 중복 작성하지 않는다.
- `pixel-canvas`(와 `grid-overlay`)에 CSS `touch-action: none`을 지정해, 손가락으로 격자 위를 드래그해 그림을 그리는 동안 브라우저의 기본 스크롤/핀치줌 제스처가 함께 발동해 그리기를 방해하지 않도록 한다. `touch-action: none`은 격자 요소에만 적용하고 헤더/툴바/팔레트 등 나머지 영역은 일반 터치 스크롤을 그대로 허용한다.
- 캔버스는 CSS `max-width: 512px; width: 100%; image-rendering: pixelated;`로 좁은 화면에서는 뷰포트 폭에 맞춰 축소 표시되고, 512px보다 넓은 화면에서는 그 이상 확대되지 않도록(흐려짐 방지) 한다. 위 4.2절의 `getBoundingClientRect()` 기반 좌표 보정 덕분에 이렇게 CSS로 축소 표시되어도 칸 클릭 판정은 항상 정확하다.
- 팔레트 스와치, 툴바 버튼은 최소 44×44px 터치 타깃을 확보한다(6절 디자인 톤에서도 동일하게 강제).

## 5. 외부 라이브러리 정책

외부 라이브러리·CDN을 사용하지 않는다. `<canvas>` API와 `<input type="color">`는 브라우저 네이티브 기능이므로 외부 의존성으로 취급하지 않는다. 격자 데이터 모델, canvas 렌더링, 포인터 입력, PNG 내보내기 모두 순수 JS로 충분히 구현 가능해 CLAUDE.md의 "외부 라이브러리 사용 최소화"와 "자체 완결" 원칙에 부합한다. 폰트도 블로그·app2048과 동일하게 시스템 폰트 스택을 사용해 웹폰트 CDN 요청 자체를 만들지 않는다.

## 6. 디자인 톤

`DESIGN-airtable.md`와 `styles/tokens.css`를 참고해 **UI 크롬(헤더, 툴바, 팔레트 컨테이너, 버튼)은 블로그 톤을 유지**하고, **격자/캔버스 작업 공간 자체는 순수 그리기 도구이므로 톤 제약에서 자유롭게 판단**한다(체크무늬 배경, 격자선 색 등은 그리기 편의를 위한 실용적 회색 계열로 별도 결정하며 signature 팔레트를 억지로 끌어올 필요 없음).

- 페이지 배경: `--color-canvas`(라이트 `#ffffff` / 다크 `#14161b`), 본문 텍스트: `--color-ink`/`--color-body`.
- "PNG로 저장" 버튼(가장 중요한 단일 CTA): 블로그의 `button-primary` 패턴 — near-black(`--color-primary`) 배경, 흰 텍스트, `rounded-lg`(12px), hover 상태 없음(no-hover 정책), `:active`만 어두워짐.
- "전체 지우기", "지우개 토글" 등 나머지 툴바 버튼: `button-secondary`(흰 배경 + `--color-hairline` 아웃라인 + `--color-ink` 텍스트) 또는 지우개처럼 토글形 버튼은 활성 시 `--color-surface-strong` 배경으로 눌림 상태를 표시.
- 팔레트 컨테이너: `rounded-md`(10px), `--color-surface-soft` 배경의 패널 안에 스와치를 그리드로 배치. 각 스와치는 `rounded-sm`(6px) 사각형, 선택된 스와치는 `--color-ink` 2px 테두리로 강조.
- 캔버스 작업 공간: `board`류 컨테이너는 `rounded-md`로 감싸되, 내부 체크무늬/격자선 색은 그리기 도구 관례상 중성 회색(라이트: 옅은 회색 두 톤 체커, 다크: 어두운 회색 두 톤 체커)으로 별도 정의 — app2048이 보드에 2048 고유 팔레트를 허용했던 것과 같은 논리로, 이 앱은 캔버스 영역에 도구 특화 색을 허용한다.
- **다크모드**: `prefers-color-scheme: dark`를 기본 신호로 앱 자체 CSS 안에서 대응(값은 `styles/tokens.css`의 다크 오버라이드를 참고해 복제). 블로그 전역 다크모드 토글(`js/theme.js`)은 공유하지 않으므로(자체 완결 원칙) OS 설정 자동 반영만 지원하고 별도 토글 버튼은 만들지 않는다 — app2048과 동일한 정책.
- 접근성/터치 타깃: 모든 버튼·스와치 44×44px 이상 확보. 키보드로도 팔레트 스와치·툴바 버튼에 포커스 이동 및 Enter/Space로 활성화 가능해야 하며(`<button>` 네이티브 요소 사용으로 기본 충족), 포커스 아웃라인은 유지한다.
