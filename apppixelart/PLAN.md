# 지침 파일 — Plan 단계 (apppixelart)

너는 my-blog 프로젝트의 새 미니 웹앱 "픽셀 아트 에디터"의 **Plan 서브에이전트**다. 이 파일이 네게 주어진 전용 지침 파일이며, 여기 명시된 범위만 수정한다.

## 배경
- 프로젝트 루트: `C:\Users\tagi7\Desktop\claude_exam\my-blog`
- 이 프로젝트의 `CLAUDE.md`를 반드시 먼저 읽어라. Plan→Build→Review→Embed→Commit 워크플로, 서브에이전트 규칙, 웹앱 규칙, 앱 디렉토리 구조, Build 서브에이전트 금지 파일 목록이 정의돼 있다.
- 참고로 먼저 만들어진 미니 웹앱 `C:\Users\tagi7\Desktop\claude_exam\my-blog\app2048\spec.md`를 읽어서 이 프로젝트의 spec.md 작성 스타일(항목 구성, 상세도)을 참고해라. (app2048은 게임이고 이번엔 에디터라 내용은 다르지만 문서 구조 관례는 따를 것.)
- 사용자 요청 원문: "픽셀아트에디터를 만들어줘. 16*16격자에 도트를 찍어서 그림을 그리고 완성하면 png로 저장할 수 있게 해줘. 색상 팔레트도 넣어줘."

## 네가 할 일
1. `C:\Users\tagi7\Desktop\claude_exam\my-blog\apppixelart\spec.md` 파일을 작성한다. 이것이 유일한 산출물이다 — **코드는 한 줄도 작성하지 않는다.**
2. spec.md에는 다음이 들어가야 한다:
   - 앱 개요: 16x16 픽셀 격자에 클릭(및 드래그)으로 도트를 찍어 그림을 그리고, 색상 팔레트에서 색을 고르고, 완성한 그림을 PNG 파일로 저장(다운로드)할 수 있는 에디터.
   - 화면 구성: 단일 화면(격자 + 색상 팔레트 + 툴바(지우개/전체 지우기/PNG 저장 버튼 등))으로 설계 가능한지 판단하고, 화면이 3개 미만이면 Build 서브에이전트를 나눌 필요가 없다고 명시. (3개 이상이라 판단되면 근거와 분리안을 제시.)
   - 파일 구조: `/apppixelart/` 폴더 안에 자체 완결. app2048의 관례(엔진/로직 모듈과 UI 컨트롤러 분리)를 참고해 합리적인 구조를 제안 (예: `index.html`, `styles.css`, `editor.js`).
   - 핵심 기능 설계:
     - 16x16 격자 렌더링 방식 (DOM 기반 div 그리드 vs `<canvas>` 방식 중 하나를 선택하고 이유를 적을 것 — PNG로 저장해야 하므로 최종적으로는 `<canvas>`의 `toDataURL('image/png')` 또는 이에 준하는 방법을 반드시 사용해야 함을 명시. DOM 격자로 그리고 저장 시점에만 canvas로 변환하는 방식도 가능하니 트레이드오프를 적을 것)
     - 클릭 + 드래그로 연속해서 칠하기(마우스다운 상태에서 이동 시 지나가는 칸에 칠해짐) 지원 여부와 방식
     - 색상 팔레트: 기본 제공 색상 목록(몇 개 정도가 적절할지 제안, 예: 12~16색 + 검정/흰색/투명(지우개) 포함) + 커스텀 색상 선택(`<input type="color">` 네이티브 컬러피커 활용 등 외부 라이브러리 없이 가능한 방법) 여부
     - 지우개 도구, 전체 지우기(Clear) 버튼
     - PNG 저장: 16x16 원본 해상도 그대로 저장할지, 확대(예: 512x512, nearest-neighbor 방식으로 픽셀이 뭉개지지 않게)해서 저장할지 제안하고 이유를 적을 것 (픽셀아트는 원본 그대로 저장하면 너무 작아서 보기 불편하므로 확대 저장을 권장하되 명시적으로 판단할 것)
     - 모바일 터치로 그리기 지원 (CLAUDE.md 웹앱 규칙 "모바일에서도 사용할 수 있어야 한다" 필수 충족 — 터치 드래그로 칠하기, 스크롤과 충돌하지 않게 처리하는 방법 포함)
   - 외부 라이브러리는 쓰지 않는 방향을 기본으로 할 것 (`<canvas>` API와 `<input type="color">`는 브라우저 네이티브라 외부 의존성 아님). CDN이 꼭 필요한 이유가 있다면 근거를 적을 것.
   - my-blog의 기존 디자인 톤(에디토리얼, 화이트/다크 캔버스, near-black 프라이머리 버튼, hover 없음, 라운드 코너 위계 — `DESIGN-airtable.md`와 `styles/tokens.css` 참고)을 UI 크롬(툴바, 버튼, 팔레트 컨테이너)에 적용하고, 격자/캔버스 자체는 순수 작업 공간이므로 톤 제약에서 자유롭게 판단해도 된다는 점을 명시. 다크모드 대응 여부도 명시(가능하면 지원 권장, app2048처럼 OS 설정 자동 반영만 하고 별도 토글은 없어도 됨).
3. Build 단계에서 넘길 `BUILD.md` 지침 파일의 초안 요약을 spec.md 끝부분에 포함한다 (실제 BUILD.md 파일 생성은 사용자 승인 후 별도 단계).
