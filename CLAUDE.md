## 프로젝트 개요
마크다운 기반 블로그+미니 웹앱 포트폴리오 HTML, CSS, Javascript만 사용

## 작업 사이클
1. Plan — 서브에이전트를 만들어 계획을 작성한다. 어떤 웹앱을 만들지, 파일구조는 어떻게 할지 정리한다. 정리한 계획은  spec.md로 저장하며, 사용자 승인을 받는다.
2. Build — 서브에이전트를 만들어 구현한다. 웹앱은/app{앱이름}/폴더로 독립적으로 만든다. 화면이 3개 이상이면 화면별로 서브에이전트를 나눈다. 블로그의 다른 파일을 건드리지 않는다.
3. Review — 별도 서브에이전트를 만들어 테스트하고 검증한다.  브라우저에서 정상 작동하는지, 코드에 문제가 없는지 확인하고 review.md를 작성한다. 문제가 있으면 수정한다.
4. Embed  — 블로그 메인 페이지(index.html)에 웹앱 카드를 추가한다. 카드에는 제목, 설명, 미리보기 이미지 또는 iframe을 넣는다.
5. Commit — 기능마다 깃 커밋하고 깃허브에 푸시한다.

## 서브에이전트 규칙
- 서브에이전트에게 작업을 넘길 때 전용 지침 파일(.md)을 만들어 전달한다.
- 서브에이전트는 지침 파일에 명시된 범위만 수정한다.
- Build 서브에이전트와 Review 서브에이전트는 반드시 분리한다.
- Build가 복잡하면 여러 서브에이전트로 나눈다. 각각 독립적으로 작업할 수 있게 범위를 겹치지 않게 나눈다.

## 웹앱 규칙
- 모든 웹앱은 /app{앱이름}/폴더 안에 자체 완결한다.
- 외부 라이브러리 사용을 최소화 한다. CDN은 허용한다.
- 모바일에서도 사용할 수 있어야 한다.

## 규칙
- 승인 없이 구현을 시작하지 않는다.
- 막히면 사용자에게 알린다.

## 앱 디렉토리 구조
모든 웹앱은 아래 구조로 `/app{앱이름}/` 폴더 안에 자체 완결한다. 최상위에 `/specs/` 같은 공용 폴더는 두지 않는다.

```
/app{앱이름}/
  spec.md              # Plan 산출물 — 무엇을 만들지, 화면 구성, 파일구조. 사용자 승인 전까지는 이 파일만 존재.
  BUILD.md              # Build 서브에이전트 지침 파일. 화면이 3개 이상이면 화면별로
                         # BUILD-{화면명}.md 로 나눈다 (예: BUILD-calculator.md, BUILD-history.md).
  review.md             # Review 서브에이전트 산출물 — 테스트 결과, 발견한 문제, 수정 내역.
  index.html             # 앱 진입점
  ...                    # 앱 자체 js/css/asset 전부 이 폴더 안에만 둔다.
```

## Build 서브에이전트 금지 파일
Build/Review 서브에이전트는 자신에게 주어진 지침 파일(`/app{앱이름}/BUILD*.md`)에 명시된 `/app{앱이름}/` 내부 파일만 만들고 고친다. 아래 파일/폴더는 절대 만들거나 고치거나 지우지 않는다:

- `index.html`, `post.html`
- `js/`, `styles/`, `posts/`, `scripts/`
- `CLAUDE.md`, `DESIGN-airtable.md`, `README.md`
- `package.json`, `.gitignore`, `.claude/`

`index.html`에 웹앱 카드를 추가하는 것은 오직 Embed 단계에서만 일어나며, Build·Review 서브에이전트의 역할이 아니다.
