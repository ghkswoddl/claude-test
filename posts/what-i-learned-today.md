# 오늘 배운 것

클로드 코드로 프레임워크 없는 블로그를 직접 만들면서, HTML·CSS·JavaScript가 각자 어떤 역할을 맡는지 다시 한번 명확하게 정리할 수 있었다.

## HTML — 구조와 의미

HTML은 콘텐츠의 **뼈대와 의미**를 담당한다. `article`, `nav`, `header`, `footer` 같은 시맨틱 태그로 페이지의 각 영역이 무엇인지 표현하고, 실제 텍스트·데이터는 JavaScript가 채워 넣는 빈 컨테이너(`#article-grid`, `#post-body` 등)를 미리 마련해두는 역할을 한다. 스타일이나 동작 로직은 최대한 HTML에 섞지 않는 게 유지보수에 유리했다.

## CSS — 디자인 시스템의 표현

CSS는 디자인 토큰을 실제 화면으로 옮기는 역할을 한다. 이번 프로젝트에서 특히 인상적이었던 점:

- 색상·간격·타이포그래피를 커스텀 프로퍼티(`--color-ink`, `--space-lg` 등)로 미리 선언해두면, 다크모드 전환이 `[data-theme]` 속성 하나로 끝난다.
- 반응형은 `min-width` 미디어쿼리 몇 개로 그리드 컬럼 수만 바꿔주는 정도로 충분했다.
- 컴포넌트마다 별도 파일로 나누니(`components.css`, `layout.css`) 어떤 스타일이 어디서 오는지 추적하기 쉬웠다.

## JavaScript — 데이터와 상호작용

JavaScript는 다음 세 가지를 담당했다.

1. **데이터 가져오기**: `fetch()`로 `posts/index.json`과 개별 `.md` 파일을 읽어온다.
2. **변환**: 마크다운 텍스트를 HTML로 바꾸고(`renderMarkdown`), 프론트매터를 파싱한다.
3. **상호작용**: 검색어 입력, 태그 필터 클릭, 다크모드 토글 같은 사용자 동작에 반응해 DOM을 다시 그린다.

> 프레임워크 없이 순수 JS만 써도, 상태를 모듈 스코프 변수 몇 개로 관리하고 `innerHTML`을 다시 그리는 패턴만으로 충분히 동작하는 앱을 만들 수 있었다.

## 정리

결국 세 언어는 각각 **구조(HTML) → 표현(CSS) → 동작(JavaScript)** 으로 역할이 뚜렷하게 나뉜다. 이 경계를 지키면서 작업하니 코드가 어디에 있어야 하는지 고민할 일이 거의 없었다.
