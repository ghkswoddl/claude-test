# 오늘 배운 것

이 블로그에 미니 웹앱을 하나둘 추가하면서, 클로드 코드를 그냥 "시키는 대로 코드 짜는 도구"가 아니라 **하네스 — 스킬 — 서브에이전트**라는 세 층으로 나눠서 쓰는 감을 잡을 수 있었다.

## 하네스 — 프로젝트를 둘러싼 규칙과 환경

하네스는 클로드가 이 프로젝트 안에서 "어떻게 일해야 하는지"를 정해두는 틀이다. 이번 프로젝트에서는 이런 것들이 하네스에 해당했다.

- **CLAUDE.md** — 작업 사이클(Plan→Build→Review→Embed→Commit), 폴더 구조, 서브에이전트가 건드리면 안 되는 파일 목록 같은 규칙을 문서로 못박아둔다. 사람이 매번 같은 설명을 반복하지 않아도, 클로드가 이 파일을 읽고 규칙을 따른다.
- **`.claude/settings.json` 퍼미션** — git 명령이나 테스트 실행처럼 자주 쓰는 안전한 작업은 미리 허용해두고, `rm -rf`나 강제 푸시처럼 되돌리기 어려운 작업은 아예 막아둔다. 매번 "이거 실행해도 될까요?"를 묻지 않아도 되면서, 정말 위험한 지점에서는 확실히 멈춘다.
- **`.claude/launch.json`** — 로컬 개발 서버를 어떻게 띄우는지 미리 등록해두면, 브라우저 미리보기가 필요할 때마다 같은 설정을 반복 입력할 필요가 없다.

즉 하네스는 "이 프로젝트에서 클로드가 안전하고 일관되게 동작하도록 만드는 환경 설정"에 가까웠다.

## 스킬 — 반복되는 절차를 재사용 가능한 형태로

스킬은 한 번 검증된 작업 절차를 다음에도 그대로 써먹을 수 있게 패키징한 것이다. 이번에 두 개를 직접 만들어봤다.

- **`webapp-blog`** — 미니 웹앱 하나를 Plan→Build→Review→Embed→Commit 순서로 만드는 절차 자체. 서브에이전트에게 넘길 지침 파일 템플릿, 스크린샷 없을 때 쓰는 CSS 목업 프리뷰 요령, 실제로 겪었던 버그 패턴(`hidden` 속성과 CSS `display`가 충돌하는 문제, Windows에서 대소문자 구분 없는 파일명 충돌)까지 다 담겨 있다.
- **`webapp-harness`** — 프로젝트 자체를 세팅하는 절차. 기술 스택 컨벤션, 퍼미션 템플릿, 그리고 제미나이로 2차 검토를 받는 절차를 정리했다.

스킬을 만들면서 가장 크게 느낀 건, **왜 이렇게 하는지 이유를 같이 적어두는 게 중요하다**는 점이다. "이렇게 해라"만 적으면 다음에 비슷하지만 조금 다른 상황을 만났을 때 응용이 안 되는데, 이유를 적어두면 판단 기준으로 쓸 수 있다.

## 서브에이전트 — 범위를 좁혀서 안전하게 위임하기

서브에이전트는 특정 작업만 하도록 범위를 좁혀서 위임하는 방식이다. 이번 프로젝트의 Build/Review 단계가 정확히 이 패턴이었다.

- Build 서브에이전트에게는 `/app{이름}/` 폴더 안의 파일만, Review 서브에이전트에게는 검증과 그 폴더 안 수정만 허용했다. 지침 파일에 "이 파일들은 절대 건드리지 마라"를 명시해두면, 작업이 끝난 뒤 `git status`로 실제로 그 범위를 지켰는지 바로 확인할 수 있다.
- Build와 Review를 **같은 서브에이전트가 아니라 별도로** 돌리는 게 중요했다. 자기가 만든 코드를 자기가 검증하면 놓치는 게 생기기 마련이라, 실제로 2048 게임에서 오버레이가 계속 떠 있는 버그를 Build/Review 둘 다 놓쳤다가 사용자가 스크린샷으로 잡아준 적이 있었다.
- 나아가 **클로드가 아닌 다른 모델(제미나이)에게 2차 검토**를 맡기니, 같은 모델 계열이 공유하는 사각지대와는 다른 종류의 문제(마크다운 렌더러의 XSS 취약점)를 잡아낼 수 있었다. 다만 다른 모델의 지적도 곧이곧대로 믿지 않고, 하나하나 코드로 검증한 뒤에 고칠지 말지 판단하는 과정이 꼭 필요했다.

> 서브에이전트에게 일을 맡길 때 "무엇을 할 수 있는지"보다 "무엇을 할 수 없는지"를 먼저 정해두면, 결과를 신뢰하기가 훨씬 쉬워진다.

## 정리

하네스는 **환경과 규칙**, 스킬은 **검증된 절차의 재사용**, 서브에이전트는 **좁은 범위로 나눈 위임**이라는 세 층으로 정리된다. 셋을 따로 두면 각각을 독립적으로 점검하고 개선할 수 있어서, 프로젝트가 커져도 관리가 덜 힘들어지는 느낌이었다.
